# FerrumDB V0.3 Backend Interface Backend Feedback

## Purpose

本文档不修改 `docs/brainstorms/v0.3-backend-interface-requirements.md` 的原始需求，也不改变前后端约定的接口意义。它只从 `src-tauri` 当前实现出发，回答三件事：

- 哪些现有后端逻辑可以直接复用
- 哪些地方需要重构或补层，但不应改变接口语义
- 哪些需求点存在实现风险、歧义或与当前后端现实不一致，应该反馈给前端再确认

## Confirmed Backend Reality

以下内容已通过 `src-tauri` 当前代码确认：

- 已存在的 Tauri command 只有：`list_connections`、`get_connection`、`create_connection`、`update_connection`、`delete_connection`、`test_connection`
- 已有 SQLite 持久化层：`src-tauri/src/storage/connection.rs`
- 已有凭据加密/解密模块：`src-tauri/src/crypto/password.rs`
- 已有 MySQL / PostgreSQL 连接抽象和 pool-backed connector：`src-tauri/src/database/traits.rs`、`src-tauri/src/database/mysql.rs`、`src-tauri/src/database/postgres.rs`
- 目前没有会话存储、没有 schema introspection、没有 table preview、没有统一错误码模型

## Reuse Opportunities

### 1. Connection CRUD can remain the foundation

原文档的连接管理接口可以继续建立在现有存储层之上，不需要推倒重来：

- `list_connections`
- `create_connection`
- `update_connection`
- `delete_connection`

这些接口背后的 SQLite 存储、密码屏蔽、环境字段、标签字段已经具备基础形态，适合作为 V0.3 的 persisted connection source of truth。

### 2. test_connection can be upgraded, not replaced

当前 `test_connection` 已经具备最关键的复用路径：

- 通过 `connection_id` 从本地存储取配置
- 解密密码
- 根据 `db_type` 分发到 MySQL / PostgreSQL
- 执行真实连接测试

V0.3 不需要重写这条链路，只需要把返回值从 `bool`/`String` 提升为结构化结果，并把错误映射从“驱动原始错误”收敛成稳定的后端错误语义。

### 3. Session can reuse the existing database module

原文档要求会话成为后端一等对象。当前代码虽然没有 session 层，但 `database` 模块已经适合做底座：

- connector 已经知道如何 connect / disconnect
- 可以继续用同一套 `ConnectionConfig`
- 可以在 `AppState` 里新增 runtime session registry，而不是改写 SQLite schema

这意味着会话层更像是“新增 orchestration layer”，而不是“重做数据库连接能力”。

## Recommended Backend Refactors

这些建议不改变原文档接口意义，只是为了让实现更稳、更可维护。

### 1. Split command DTOs from storage models

当前命令层大量直接复用 `storage::ConnectionConfig`。V0.3 如果继续扩展 session / Data Browser，很容易把存储结构、命令返回结构和未来 IPC contract 混在一起。

建议：

- 保留 `storage` 里的持久化模型
- 在 `commands` 或新建 `contracts` 模块中定义 command-facing DTO
- 让 `Connection`、`Session`、`SchemaListResponse`、`TablePreviewResponse` 都走显式映射

这样可以在不改前端接口语义的前提下，减少后续结构漂移。

### 2. Introduce a unified backend error type

原文档要求结构化错误，但当前命令层基本是 `Result<_, String>`。如果继续沿用字符串拼装，实现会很快失控。

建议：

- 新增统一 `AppError` 或 `CommandError`
- 由 storage / crypto / database 错误收敛映射到稳定 `code`
- 由命令层统一序列化成前端所需错误对象

这是满足 R6、R19-R24 的最小可持续做法。

### 3. Add a runtime session registry to AppState

原文档要求 `open_connection_session`、`close_connection_session`、`get_session_overview`，而当前 `AppState` 只有 `storage: Mutex<Storage>`。

建议：

- 在 `AppState` 内新增 session registry
- registry 维护 `session_id -> runtime session`
- runtime session 内持有连接元信息、访问模式和底层数据库连接对象

这既符合会话是一等对象的需求，也不会污染现有连接存储逻辑。

### 4. Separate introspection queries from generic connection code

`list_session_schemas` 和 `get_table_preview` 不建议直接塞进现有 `mysql.rs` / `postgres.rs` 的连接 trait 中，否则 trait 很快会混合“连接生命周期”和“元数据浏览”两类职责。

建议：

- 保持现有 connector 聚焦连接能力
- 新增 metadata / browser service 层
- 在该层中按数据库类型实现 schema introspection 和 table preview

这样更便于后续扩展 SQL Editor，而不把所有行为都堆进一个巨型 trait。

## Requirement-by-Requirement Feasibility Notes

### Connection Inventory Compatibility

- R1-R3：当前后端基本可满足，主要是保持兼容并避免把 session 字段混入 persisted connection 返回
- R4：当前后端不能满足。`delete_connection` 命令层目前只返回 `Result<(), String>`，且不会表达 session invalidation 语义

### Connection Test and Session Lifecycle

- R5：当前接口方向正确，可复用现有 `test_connection`
- R6：当前不能满足，需要结构化返回和错误映射
- R7-R11：当前都没有实现，需要新增会话层

### Data Browser Landing

- R12-R18：当前都没有实现，但可以建立在新增 session registry + 数据库 metadata 查询层之上

### Error Model and State Semantics

- R19-R24：当前无法直接满足，需要统一错误类型和命令返回契约

### Security and Environment Framing

- R27：当前读取型连接接口已具备密码屏蔽基础
- R25、R26、R28：目前没有 session payload，因此还未实现；但可以通过会话层补齐

## Feedback To Frontend / Requirements Owners

下面这些不是要求后端擅自改接口，而是建议明确反馈给前端或需求文档维护者确认。

### 1. `delete_connection` response shape needs an explicit compatibility decision

原文档要求：

- `{ deleted: true, invalidated_session_id?: string }`

当前后端现实是：

- 命令层返回 `()`
- 且当前不会区分“删除成功”与“连接不存在”

需要前端/需求侧明确：

- 是否允许把 `delete_connection` 的返回结构从 `void` 升级为对象
- 如果前端当前已经假设 `void`，是否接受联调前做一次兼容调整

这是最值得尽早对齐的接口点。

### 2. `test_connection` parameter naming should be locked

原文档推荐请求是：

- `{ connection_id }`

当前后端命令是：

- `test_connection(id: String, ...)`

这不影响能力，但会影响 IPC 参数形状。建议前端和后端尽快确认是否统一为 `connection_id`，避免联调时出现“语义一致、调用签名不一致”的低级阻塞。

### 3. `row_count_label` may be expensive or inconsistent across databases

原文档把 `row_count_label` 作为 schema listing payload 的推荐字段之一。这个字段不是不能做，但要提醒前端：

- MySQL / PostgreSQL 的可得性和精度不完全一致
- 精确行数可能昂贵
- 估算值与真实值之间可能有产品解释成本

建议前端确认：

- 该字段在 V0.3 是否必须首版就展示
- 是否接受“可选字段”或“估算标签”

### 4. `access_mode` authority needs product clarification

原文档要求后端返回权威的只读/高风险标记。当前后端现实里：

- 已保存连接只有 `environment`
- 没有数据库真实权限探测逻辑

建议前端/需求侧明确：

- V0.3 的 `access_mode` 是否允许先基于环境和本地策略推导
- 还是必须等数据库真实权限探测后才算满足需求

如果不澄清，这里非常容易在“展示语义”和“真实权限”之间产生误解。

### 5. Session identifier scope should be fixed before implementation starts

原文档已经把这列为未决问题，这里补充它对实现的直接影响：

- 如果是进程内短生命周期 ID，实现简单，适合现有 `AppState`
- 如果要求跨窗口 / 跨重启恢复，现有模块边界和存储设计都要扩展

建议尽快锁定，不然会影响 session registry 设计。

## Suggested Delivery Strategy

如果目标是不破坏原文档语义、同时最大化复用现有后端，建议按下面顺序实现：

1. 先升级 `test_connection` 为结构化结果，但保留“测试已保存连接”这一能力边界
2. 引入 runtime session registry，补齐 `open_connection_session` / `close_connection_session` / `get_session_overview`
3. 在 metadata service 层实现 `list_session_schemas`
4. 实现 `get_table_preview`
5. 最后处理 `delete_connection` 与 session invalidation 的联动

这个顺序能最大程度复用已有 storage / crypto / database 模块，也能尽量减少一次性大重构。

## Simple Frontend Feedback

基于当前前端已实现状态，这份后端反馈里的大多数建议都不需要立刻触发前端修改。

- 现在不建议为了后端内部重构而改前端
- 真正值得改前端的时机，是后端真实提供 `test_connection`、session 接口和 Data Browser 接口之后
- 在那之前，前端继续保持当前 mock 方案即可，避免为尚未落地的后端形态提前付出改造成本
