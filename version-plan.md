# FerrumDB 版本计划（重构版）

> 更新日期: 2026-04-08
> 参考资料:`stitch/*`

---

## 新的拆分原则

1. 先做单机单人闭环，再做高级能力。
2. 先做只读能力，再做写入/变更能力。
3. 先让工作台成形，再叠加 AI、协作、插件。
4. 多数据库支持按数据库类型分批上线，不再打包成一个版本。
5. 每个版本都要能明确映射到一个 `stitch` 设计稿页面。

---

## 设计稿到版本的映射

| 设计稿 | 主要能力 | 建议落点 |
|------|----------|---------|
| `stitch/connections_dashboard` | 连接工作台、全局导航、搜索、连接卡片 | `V0.2` ~ `V0.3` |
| `stitch/advanced_sql_editor` | SQL 编辑器、多标签、结果区、AI 入口 | `V0.4` |
| `stitch/data_browser_editor` | Schema 树、表格浏览、筛选、分页、批量操作 | `V0.5` ~ `V0.7` |
| `stitch/settings_preferences` | 偏好设置、编辑器设置、安全开关 | `V0.6` |
| `stitch/explain_visualizer_fixed_connections` | Explain 可视化、节点详情、性能建议 | `V0.8` |
| `stitch/visual_er_diagram` | ER 浏览、关系线、属性面板、变更区 | `V0.9` ~ `V1.0` |
| `stitch/syntactic_flux/DESIGN.md` | 统一暗色设计语言、颜色/字体/间距约束 | 贯穿全部版本 |

## 新版本路线图

### V0.1 - 基础骨架与本地连接管理

**目标**: 应用能启动，用户能保存和管理连接配置。

**核心任务**
- Tauri 2.x + React + TypeScript 基础工程稳定
- SQLite 本地存储连接配置
- 密码加密存储与基础安全校验
- 连接卡片 CRUD、空状态、删除确认、基础 Toast

**对应设计稿**
- `stitch/connections_dashboard`

**版本完成标准**
- 用户可新增、编辑、删除连接配置
- 重启后配置仍然存在
- 密码不以明文暴露给前端列表接口

**暂不包含**
- 实际连接数据库
- SQL 执行
- 路由化多页面工作台

---

### V0.2 - 工作台外壳与连接首页定稿

**目标**: 先把产品壳层做对，形成一致的桌面工作台体验。

**核心任务**
- 统一侧边栏、顶部栏、状态栏、主内容区布局
- 补齐模块入口占位: Connections / SQL Editor / Data Browser / ER Diagram / History / Settings
- 连接首页视觉对齐 `stitch/connections_dashboard`
- 搜索、环境筛选、标签/分组展示
- 预留全局 Command Palette、通知位、用户操作位

**对应设计稿**
- `stitch/connections_dashboard`
- `stitch/syntactic_flux/DESIGN.md`

**版本完成标准**
- FerrumDB 从“单页 CRUD”变成“多模块工作台”
- Connections 首页达到可演示级视觉完成度
- 新页面具备统一壳层，不再重复搭布局

**暂不包含**
- 真正进入 SQL/Data/ER 页面业务逻辑
- SSH 隧道

---

### V0.3 - 真实数据库连接与会话进入

**目标**: 从“保存配置”升级到“真正连上数据库并进入工作区”。

**核心任务**
- MySQL / PostgreSQL 连接测试打通
- 连接状态展示: connected / failed / reconnecting
- 连接成功后进入会话上下文
- 会话级别的当前连接信息、环境标识、只读标识
- 错误反馈和失败重试机制

**对应设计稿**
- `stitch/connections_dashboard`
- `stitch/advanced_sql_editor`
- `stitch/data_browser_editor`

**版本完成标准**
- 用户能从连接卡片发起连接测试或进入工作区
- 成功连接后，SQL Editor / Data Browser 能拿到当前连接上下文
- 明确区分开发、预发、生产环境状态

**暂不包含**
- SSH Jump Server
- 查询编辑高级能力

---

### V0.4 - SQL 编辑器 MVP

**目标**: 形成最小可用查询体验。

**核心任务**
- SQL 编辑区、查询标签页、执行按钮
- 单条 SQL 执行、选中执行、查询结果表格
- 查询错误定位和执行状态反馈
- 基础历史记录
- 为后续自动补全和 AI 助手预留侧边入口

**对应设计稿**
- `stitch/advanced_sql_editor`

**版本完成标准**
- 用户可以连库后编写并执行 SQL
- 查询结果能在当前工作区稳定展示
- 基础历史可回看最近查询

**暂不包含**
- AI 生成 SQL
- Explain 树图
- 高级 LSP

---

### V0.5 - 数据浏览器 MVP

**目标**: 先把“只读浏览”做好，再谈编辑。

**核心任务**
- Schema / 表树浏览
- 表数据分页展示
- 列信息、基础排序、基础筛选
- 大表加载与虚拟滚动策略
- 与 SQL Editor 共用连接上下文

**对应设计稿**
- `stitch/data_browser_editor`

**版本完成标准**
- 用户能浏览数据库、表、字段和分页数据
- 大表场景下 UI 不明显卡顿
- 从连接首页到数据浏览主链路闭环

**暂不包含**
- 行编辑
- JSON 层级编辑
- 批量修改

---

### V0.6 - 设置中心与生产安全

**目标**: 在开放更多写能力之前，先补齐安全和偏好控制。

**核心任务**
- 设置页框架和分类导航
- 编辑器偏好设置: 字体、Tab Size、主题模式
- 安全设置: 生产环境只读模式、危险操作确认
- 连接安全策略集中管理
- 本地偏好持久化

**对应设计稿**
- `stitch/settings_preferences`

**版本完成标准**
- 设置页可独立进入并持久保存用户偏好
- 生产环境可以被显式标记为只读/高风险
- 后续写操作能力具备安全前置条件

**暂不包含**
- 云同步
- 团队级策略分发

---

### V0.7 - 数据编辑与 JSON 支持

**目标**: 在安全保护前提下开放数据修改能力。

**核心任务**
- 单元格编辑、增删行、批量操作
- 编辑前后校验与回滚提示
- JSON / JSONB 字段编辑器
- 写操作确认与只读模式联动
- 与查询结果表格复用部分表格能力

**对应设计稿**
- `stitch/data_browser_editor`

**版本完成标准**
- 用户可对非只读连接进行基础数据编辑
- JSON 字段不再只能当纯文本处理
- 写入失败时 UI 能正确提示并保持一致状态

**暂不包含**
- 结构变更
- Migration 导出

---

### V0.8 - Explain 可视化分析

**目标**: 从“能跑 SQL”升级到“能理解 SQL 为什么慢”。

**核心任务**
- `EXPLAIN / EXPLAIN ANALYZE` 结果解析
- 节点树图展示、连线布局、节点详情面板
- 运行时间、成本、扫描类型等关键指标展示
- 从 SQL Editor 一键进入 Explain 视图

**对应设计稿**
- `stitch/explain_visualizer_fixed_connections`

**版本完成标准**
- 用户能从查询结果切换到 Explain 可视化
- 至少覆盖 PostgreSQL，MySQL 作为兼容扩展
- 图视图和原始计划文本可互相切换

**暂不包含**
- AI 优化建议
- 自动索引推荐

---

### V0.9 - ER 图浏览与导出

**目标**: 先做只读的结构洞察，不急着做图形化改表。

**核心任务**
- 从数据库 schema 自动生成 ER 图
- 表、字段、关系线展示
- 拖拽、缩放、定位
- 导出 SVG / PNG

**对应设计稿**
- `stitch/visual_er_diagram`

**版本完成标准**
- 用户能为 MySQL / PostgreSQL 生成 ER 图
- 基础拖拽缩放流畅可用
- 可导出静态图用于评审或文档

**暂不包含**
- 图形化改表
- 迁移脚本生成

---

### V1.0 - 结构设计与 Migration 导出

**目标**: 在只读 ER 图稳定后，再开放结构设计能力。

**核心任务**
- 图形化编辑表/字段/索引关系
- 生成 `ALTER TABLE` 预览
- 导出 migration 文件
- 对接常见格式: `golang-migrate`、`Alembic`

**对应设计稿**
- `stitch/visual_er_diagram`

**版本完成标准**
- 用户可以在图形界面提出结构变更
- 每次变更都能看到可审查的 SQL / Migration 输出
- 不直接静默改库，始终保持“预览优先”

---

### V1.1 - AI SQL 助手

**目标**: AI 不再单独成一个大方向，而是建立在已成熟的查询工作流上。

**核心任务**
- 自然语言转 SQL
- SQL 解释与结果说明
- Explain 结果智能总结
- 可配置模型源: 云端 / 本地模型

**依赖前置**
- `V0.4` SQL 编辑器
- `V0.8` Explain 可视化
- `V0.6` 设置中心

**版本完成标准**
- AI 能嵌入已有 SQL 工作流，而不是成为孤立入口
- 用户可明确知道 AI 的数据边界和模型来源

---

### V1.2 - 多数据库支持一期

**目标**: 先扩到最贴近当前架构的数据库类型。

**核心任务**
- SQLite 本地数据库浏览与查询
- 文件型数据库打开/关闭流程
- 与已有 SQL / Data Browser 页面共享大部分交互

**版本完成标准**
- SQLite 成为第 3 个一等公民数据库类型

---

### V1.3 - 多数据库支持二期

**目标**: 引入偏分析型和非事务型场景，但不强求统一体验一步到位。

**核心任务**
- Redis: 键空间浏览、基础值查看
- ClickHouse: 查询与结果浏览

**版本完成标准**
- 每类数据库至少有一个清晰、最小但可信的核心使用场景

**暂不包含**
- Redis 实时监控大盘的完整形态

---

### V1.4 - 多数据库支持三期

**目标**: 最后补充文档型数据库。

**核心任务**
- MongoDB 文档浏览
- 基础查询与聚合执行
- 文档结构树形查看

**版本完成标准**
- MongoDB 有独立可用的浏览与查询体验

---

### V2.0 - 团队协作与插件生态

**目标**: 等单机产品稳定后，再做真正的生态层。

**核心任务**
- 云端同步
- 团队共享连接配置
- 插件 API
- 扩展面板或自定义工具能力

**说明**
- 原计划里的“协作、插件、监控仪表盘”保留到这个阶段是合理的
- 但前提是前面的单机工作流已经非常稳，否则会把复杂度放大

---

## 推荐开发顺序

建议按下面 4 个阶段推进，而不是盯着“大版本号”推进：

1. **基础闭环阶段**
   - `V0.1` → `V0.3`
   - 目标: 能保存连接、能连库、能进入工作区

2. **核心使用阶段**
   - `V0.4` → `V0.7`
   - 目标: 能查、能看、能改，并且有基本安全控制

3. **专业分析阶段**
   - `V0.8` → `V1.0`
   - 目标: 能分析执行计划、能看结构、能导出迁移

4. **扩展增强阶段**
   - `V1.1` → `V2.0`
   - 目标: AI、多数据库、协作、插件
